# LP AI Builder データ設計書

## 1. 文書概要

### 1-1. 目的

本書は、LP AI Builder におけるデータ構造を定義するための設計書である。主に以下を対象とする。

- React フロントエンドで扱う状態データ
- PHP バックエンドで受け取るAPI入出力データ
- 永続化対象となるDBテーブル構造
- コード生成・JSONエクスポートに使用する内部データ形式

本システムは**制作ツール寄り**であるため、単なるLP完成データだけでなく、**編集中の状態・セクション単位の設定・生成履歴**を保持できる構造とする。

---

## 2. データ設計方針

### 2-1. 基本方針

- フロントエンドでは **1プロジェクト = 1つの統合オブジェクト** を基本とする
- 表示順を持つデータは配列で管理する
- 可変項目の多い構造は `JSON` または `TEXT(JSON文字列)` で保持可能にする
- 将来のテンプレート機能・複製・A/B案生成を見越して拡張しやすい設計にする
- Reactのstate構造とAPIレスポンス構造をできるだけ近づける

### 2-2. データ分類

本アプリで扱うデータは以下に分類する。

1. プロジェクト基本情報
2. デザイン設定
3. セクション設定
4. 繰り返し要素（特徴、FAQ、価格表、レビューなど）
5. AI生成結果
6. コード生成結果
7. UI補助状態
8. エクスポート用データ

## 10. 保存単位の考え方

### 10-1. 推奨保存方式

初期実装では以下の2案がある。

#### 案A: 正規化寄り

- projects
- project_designs
- project_sections
- project_items
- project_links
- project_assets

**メリット**

- 検索しやすい
- 画面ごとの更新がしやすい
- 将来機能追加しやすい

**デメリット**

- 実装コストが上がる

#### 案B: 半JSON寄り

- projects に主要項目
- detail_json にまとめて保存

**メリット**

- 実装が早い
- Reactのstate構造をそのまま保存しやすい

**デメリット**

- 検索や集計に弱い
- 一部更新が面倒

### 10-2. 推奨結論

本アプリは制作ツール寄りで将来拡張も多いため、
**主要部分は正規化、可変部分はJSONで補助** するハイブリッド方式を推奨する。

## 11. バリデーション観点

### 11-1. 基本情報

- projectName: 必須、255文字以内
- productName: 必須、255文字以内
- summary: 必須
- ctaUrl: URL形式

### 11-2. デザイン情報

- カラーコード: `#RRGGBB` 形式
- radiusLevel: 0以上
- shadowLevel: 0以上

### 11-3. セクション情報

- sortOrder: 1以上
- sectionKey: 定義済みキーのみ
- enabled: boolean

### 11-4. 繰り返し項目

- FAQは質問・回答セット
- pricingはプラン名必須
- testimonialはコメント必須

## 15. 結論

LP AI Builder のデータ設計は、**制作中の柔軟な編集**と**将来的な拡張性**の両方を重視して構成する。

特に重要なのは以下。

- フロントでは統合オブジェクトで扱う
- バックでは主要データを正規化する
- 可変項目はJSONを併用する
- コード生成前に出力用データへ整形する

この設計により、React + PHP 構成でも扱いやすく、実装しながら機能追加しやすい基盤を作ることができる。
